This is the test page for WMPlayerButtonElement.

Prior to this custom element's creation, WMPlayerElement would spawn ordinary <button> and <input> elements and style them with pseudo-elements. This had its benefits and its downsides, compared to these custom elements.

Details

Advantages of native elements

  • Significantly less JavaScript needed to achieve the same behaviors, for basic widgets like buttons. (This would be far, far, far less of an issue if Safari weren't refusing to support customized built-in elements in favor of custom attributes, a thing which (though cool) doesn't actually accomplish all the use cases of customizing built-ins.)

  • Less fiddling with ARIA needed to achieve basic accessibility, and less custom element jank in general.

Disadvantages of native elements

  • I'd like to make it possible to configure the player's controls and layout using slots. However, selectors like ::slotted(foo)::before don't work: you can't access pseudo-elements of a slotted object. This means that if you place a bare <button> inside of a pseudo-element, we can't re-skin that button with CSS alone.

    If the button is a custom element, then we can re-skin it however we like "from inside," allowing us to slot it into the player. Of course, since Firefox and Safari are both refusing to implement :host-context(), styling the buttons "from inside" is more limited than styling them "from outside," which is particularly painful for things like recoloring their label glyphs based on whether player controls are being overlaid on the played media.

  • Some of our buttons have non-rectangular shapes. In order to give them non-rectangular hitboxes, we have to make them zero-size along at least one axis, and then use pseudo-elements to split their hitbox apart from their visuals; we can then apply a clip-path to the hitbox. This, however, leads to an ever-growing tower of hacks.

    Making the button zero-size means that it doesn't position itself properly within flex layouts (with respect to things like centering). We can remedy this by giving the button margins. Unfortunately, a zero-size button also has a zero-size focus ring, and the focus ring doesn't encompass the button's margins, so we have to replace the focus ring. Focus rings are also subject to clip-paths, however, so we can't always move the focus ring to the hitbox; we have to settle for moving it to the graphics.

    (It would be possible to solve this problem if we could set pointer-events: none on the button and then set pointer-events: auto on the hitbox; the button then wouldn't need to be zero-size. I need to look into this.)

    By contrast, a custom-element button can define separate child elements for the hitbox, the button frame, and the button's label glyph. We can make the hitbox the actual interactive element, rather than the button itself, suppressing click events that hit the button but not the hitbox.

Improvements possible with both custom and native elements

These should be looked into, and I should try implementing them with the current native-element-based styles, before we commit to custom elements.

  • Better compartmentalization of button styles. In general it'd be really nice if we could move the button re-skin styles out of the main stylesheet for the player, to make the latter easier to maintain.

  • We should try the pointer-events: auto trick mentioned above in order to avoid zero-size buttons and the associated jank. If it works there, then that's one less advantage of a custom-element button.
